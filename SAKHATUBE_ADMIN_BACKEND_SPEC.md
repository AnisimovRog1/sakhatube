# SakhaTube Studio и backend — контракт первой боевой версии

Дата: 14 июля 2026.

## Что уже есть и чего пока нет

`admin.html` работает в двух режимах. Без токена это интерактивный локальный предпросмотр: данные живут в `localStorage`, а выбранный в демо файл не покидает устройство. После подключения короткоживущего Studio API-токена контент, витрина, модерация и защищённые multipart-загрузки работают с сервером и журналом аудита. Production-вход сотрудников через OIDC/MFA остаётся отдельным обязательным этапом.

Поэтому GitHub Pages может быть витриной продукта и прототипом Studio, но **не может быть боевой админкой**. Боевую Studio нужно разворачивать на отдельном закрытом домене после подключения серверной части.

## Опорная схема

```text
Зрительское приложение / Studio
              │
              ▼
       API gateway + rate limits
       ├── OIDC / сессии / MFA / RBAC
       ├── каталог и витрина
       ├── права доступа и покупки
       ├── комментарии и модерация
       └── аналитика
              │
   ┌──────────┼─────────────────┐
   ▼          ▼                 ▼
PostgreSQL  очередь задач   объектное хранилище
   │          │                 │
аудит      скан → транскодирование → HLS/CMAF
                             │
                             ▼
                    CDN со signed URL / DRM
```

## Роли и границы доступа

| Роль | Может | Не может |
| --- | --- | --- |
| `superadmin` | Управлять ролями, правами, публикацией и аварийным снятием с показа | Удалять неизменяемый аудит |
| `content_editor` | Создавать карточки, загружать исходники, собирать витрину, отправлять на проверку | Давать доступ к оплате, менять роли |
| `legal_reviewer` | Подтверждать паспорт прав по номеру юридической проверки | Публиковать материал, менять редакционные карточки или роли |
| `moderator` | Одобрять, скрывать и удалять комментарии; работать с жалобами | Публиковать видео, смотреть платёжные данные |
| `support` | Видеть безопасный статус доступа, устройств и обращений | Смотреть токены, исходники и платёжные реквизиты |
| `analyst` | Читать агрегированные отчёты | Менять контент, экспортировать персональные данные без отдельного разрешения |

Все изменения с публикацией, снятием с показа, правами и ролями записываются в audit log: кто, когда, что было, что стало, причина, IP и ID запроса.

## Жизненный цикл видео

1. Редактор создаёт карточку в статусе `draft` и получает одноразовый signed URL для загрузки.
2. Сервер принимает уведомление о завершении, проверяет тип и размер файла, запускает антивирусную проверку и транскодирование.
3. Конвейер создаёт HLS/CMAF-профили, постер, превью и дорожки субтитров. До успеха файл не имеет публичного URL.
4. Редактор проверяет эпизод, языки, возраст, территорию, права и цену. Затем отправляет на review.
5. Уполномоченная роль публикует сейчас или по расписанию. Сервер выдаёт поток только после проверки entitlement зрителя.
6. Снятие с показа мгновенно прекращает выдачу новых signed URL; событие и причина остаются в аудите.

## Минимальные сущности

- `content`: сериал, фильм, трейлер или клип; название, синопсис, жанры, рейтинг, языки, статус и права.
- `season`, `episode`: порядок, дата, доступ, субтитры и связь с медиафайлом.
- `media_asset`: исходник, версии обработки, длительность, контрольная сумма, статус сканирования и транскодирования.
- `home_slot`: блок, позиция, расписание и вариант витрины.
- `comment` и `moderation_action`: текст, автор, статус, жалоба, решение, причина и модератор.
- `playback_event`: анонимизированный ID сессии, запрос, первый кадр, пауза, буферизация, ошибка и завершение.
- `entitlement`: право пользователя на бесплатный, подписочный или разовый доступ. Только сервер определяет доступ к потоку.

## Первый API-контракт

```text
POST   /v1/admin/assets/upload-init
POST   /v1/admin/assets/{assetId}/upload-complete
GET    /v1/admin/assets/{assetId}

GET    /v1/admin/content
POST   /v1/admin/content
PATCH  /v1/admin/content/{contentId}
POST   /v1/admin/content/{contentId}/submit-review
POST   /v1/admin/content/{contentId}/publish
POST   /v1/admin/content/{contentId}/unpublish

GET    /v1/admin/home/slots
PATCH  /v1/admin/home/slots

GET    /v1/admin/comments?status=pending
POST   /v1/admin/comments/{commentId}/approve
POST   /v1/admin/comments/{commentId}/hide

GET    /v1/admin/analytics/overview?from=&to=
POST   /v1/events/playback
```

Каждый административный метод проверяет роль на сервере, добавляет request ID и создаёт аудит. Клиент никогда не подставляет роль, цену, право просмотра или итоговые метрики.

## Как считать цифры без самообмана

- **Просмотр**: уникальная сессия, которая получила первый кадр и достигла согласованного порога просмотра. Точный порог утверждается продуктом до запуска и одинаков для всех отчётов.
- **Лайк**: одна активная реакция на пользователя и объект; повторный запрос идемпотентен.
- **Комментарий**: виден зрителям только после проверки статуса и правил автора.
- **Первый кадр**: время от намерения смотреть до реально отображённого кадра; отдельно от времени ответа API.
- **Буферизация**: суммарная длительность и число событий после первого кадра. Ошибки и повторы не смешиваются с успешными просмотрами.

## Обязательная защита до первой оплаты

- отдельный домен Studio, HTTPS, OIDC, MFA сотрудников и короткая сессия;
- RBAC на сервере, запрет публичных бакетов и одноразовые signed URL для загрузки;
- проверка типа/размера файла, malware scan, rate limits, WAF и лимиты на комментарии;
- HLS через короткоживущие signed URL, DRM или защита потока по утверждённой модели прав;
- секреты только в менеджере секретов, не в JavaScript, репозитории или скриншотах;
- резервное копирование PostgreSQL и объектного хранилища, регулярная проверка восстановления;
- журнал инцидентов, мониторинг транскодирования, API, первого кадра и ошибок CDN;
- независимый пентест перед открытием платного контента.

## Критерий готовности к первой боевой Studio

1. Сотрудник с MFA загружает тестовый мастер-файл без прямого доступа к хранилищу.
2. Сервер завершает обработку, редактор добавляет субтитры и отправляет запись на review.
3. Разрешённый сотрудник публикует и снимает с показа эпизод; оба действия видны в аудите.
4. Зритель с правом получает HLS, без права получает понятный paywall. Ссылка на поток не работает после истечения срока.
5. Studio показывает сверяемые просмотры, лайки, комментарии и показатели плеера из серверных событий.

До выполнения этих пунктов Studio остаётся прототипом интерфейса, а не обещанием защищённой инфраструктуры.
