# Viewer-auth: контракт для Android

Этот документ — обязательный интерфейс перед включением входа в Android-клиенте.
Серверная база уже реализует `POST /v1/auth/register`, `POST /v1/auth/login` и
`GET /v1/auth/me`; до завершения e-mail verification и refresh-token rotation
Android должен оставаться в гостевом режиме.

## Разрешённый сценарий

1. Пользователь выбирает «Войти» или «Создать аккаунт».
2. Клиент отправляет e-mail либо номер телефона только по HTTPS на сервер
   SakhaTube. Пароль и одноразовый код никогда не попадают в логи и аналитику.
3. Сервер подтверждает владение адресом кодом/ссылкой, применяет лимиты попыток
   и создаёт короткую сессию.
4. Клиент получает краткоживущий access token и refresh token с ротацией.
   Пароль не сохраняется. Refresh token хранится только в защищённом Android
   Keystore-хранилище, не в резервной копии и не в обычных настройках.
5. При выходе или удалении аккаунта приложение очищает локальную сессию, а сервер
   отзывает refresh token для устройства.

## Минимальные серверные маршруты

Названия можно адаптировать, но смысл и защита обязательны:

| Маршрут | Назначение | Требования |
| --- | --- | --- |
| `POST /v1/auth/register` | Создать парольный аккаунт | уже есть: HTTPS, scrypt на сервере и rate limit; e-mail verification ещё требуется |
| `POST /v1/auth/login` | Войти по e-mail и паролю | уже есть: нейтральная ошибка и rate limit |
| `POST /v1/auth/start` | Начать passwordless/verification flow | следующий этап: нейтральный ответ без раскрытия существования аккаунта |
| `POST /v1/auth/verify` | Проверить одноразовый код/ссылку | следующий этап: одноразовый код, короткий TTL, ограничение попыток |
| `POST /v1/auth/refresh` | Ротация сессии | rotation, отзыв старого refresh token, привязка к устройству |
| `POST /v1/auth/logout` | Отзыв текущей сессии | требует действующий access token |
| `GET /v1/me` | Профиль текущего зрителя | только необходимые поля |
| `DELETE /v1/me` | Удаление аккаунта | повторное подтверждение владения аккаунтом и понятный срок обработки |

## Что должен вернуть сервер

Успешный ответ на проверку и refresh возвращает только:

```json
{
  "accessToken": "short-lived JWT or opaque token",
  "accessExpiresAt": "2026-07-16T12:00:00Z",
  "refreshToken": "rotating opaque token",
  "viewer": { "id": "viewer_…", "displayName": "…" }
}
```

Токен не передаётся в URL, query-параметрах, crash-отчётах или логах. Для
сессионных cookie применяются `Secure`, `HttpOnly`, `SameSite` и защита от CSRF;
для bearer-token API нужен `Authorization: Bearer` только по HTTPS.

## Условия включения в релиз

- проведена проверка API и негативных сценариев: неверный/просроченный код,
  повторный код, logout, украденный/отозванный refresh token;
- добавлены privacy-тексты с реальным оператором, целями, сроками и обработчиками;
- Data safety в Google Play и App Privacy в App Store отражают фактические данные;
- включён безопасный серверный rate limit, журнал безопасности без токенов,
  мониторинг подозрительных входов и поддержка удаления аккаунта;
- авторизация проверена на реальном устройстве и после обновления приложения.

До выполнения этих условий в Android UI нельзя показывать успешный вход, профиль
пользователя, покупки или персональную библиотеку.
